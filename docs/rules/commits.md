# 커밋 메시지 규칙 (Commit Rules)

협업 및 변경 이력 추적성 향상을 위해 프로젝트의 모든 Git 커밋 메시지는 아래 규칙을 적용해 작성합니다.

---

## 1. 제목 구조 (Commit Subject)
커밋의 성격을 직관적으로 구분하기 위해 명시적인 머리말 접두사를 부착합니다.

```
[접두사]: 작업 내용 요약
```

### 1-1. 허용되는 접두사 (Commit Prefixes)
* **`feat`**: 신규 기능 추가 (예: `feat: 비디오 편집기에 트랜션 레이어 추가`)
* **`fix`**: 버그 해결 및 수정 (예: `fix: AI 키 복호화 점분할 파싱 오류 수정`)
* **`docs`**: 문서 파일 추가/변경 (예: `docs: 프로젝트 구조 및 스택 정의`)
* **`refactor`**: 동작 변경 없는 코드 리팩토링 (예: `refactor: manuscripts Zustand 스토어 분할`)
* **`style`**: 코드 포맷팅, 세미콜론 누락 정돈 (로직 수정 없음)
* **`test`**: 테스트 코드 가동 및 정비

---

## 2. 권장 규칙
* **명료함**: 커밋 메시지의 제목만 보고도 코드의 어떤 영역(Studio, Admin 등)이 왜 변경되었는지 알 수 있도록 간결하고 핵심적인 키워드를 사용해 한글로 서술합니다.
* **이슈 바인딩**: 특정 이슈 해결 커밋일 시 괄호 및 이슈 번호 기입을 권장합니다.
